Continuous Delivery.


by Sebastian Isaza G. (@sebasig)
Que es Continuous delivery (CD)
Es un modelo informático propuesto inicialmente por Martin Fowler que consiste en hacer integraciones automáticas de un proyecto lo más a menudo posible para así poder detectar fallos cuanto antes.
Esta aproximación ayuda a reducir costos, tiempos y los riesgos que implica el despliegue de los cambios realizados por el equipo de desarrollo, permitiendo así realizar actualizaciones de las aplicaciones de forma incremental en ambientes productivos.
Ventajas del Continuous delivery (CD)
  • -Los desarrolladores pueden detectar y solucionar problemas de integración de forma continúa, evitando el caos de última hora cuando se acercan las fechas de entrega.
Ventajas del Continuous delivery (CD)
  • -Disponibilidad constante de una versión para pruebas, demos o lanzamientos anticipados.
  • -Ejecución inmediata de las pruebas unitarias y funcionales
  • -Monitorización continúa de las métricas de calidad del proyecto.
Principios del Continuous delivery (CD)
1- El proceso de despliegue y liberación del software se debe poder repetir y debe de ser confiable. Puede sonar tonto, pero cuando uno toma una actividad y se realiza de forma repetitiva, normalmente esto lleva a mejorar el proceso y a automatizarlo, convirtiendo esta actividad en algo simple
Principios del Continuous delivery (CD)
2- Automatizar todo lo automatizable. Por ejemplo, un despliegue manual nunca se puede considerar como una actividad que es repetible y confiable, dado a que se pueden presentar errores humanos, este se mitiga cuando es posible automatizarlo ya que es un proceso continuo que se repite de la misma forma siempre.
3- Si algo es difícil y complejo, entonces esta actividad se debe de realizar más a menudo.
Principios del Continuous delivery (CD)
4- Mantener todo en un control de versiones.
5- Listo (Done de acuerdo a las metodologías agiles), significa liberado. Esto significa que un producto no está listo hasta que es entregado al usuario funcionando de forma adecuada.
Principios del Continuous delivery (CD)
6- Construir con calidad. Un proyecto con buenas métricas de calidad (enfocadas a los puntos importantes del mismo como lo son la cobertura de pruebas unitarias, las buenas prácticas en código, violación de reglas, entre otros) demuestra que un código puede ser más mantenible que uno que no posea estas métricas.
Principios del Continuous delivery (CD)
7- Todos son responsables del proceso de despliegue. Un proyecto desplegado en el equipo de un desarrollado no va a generar ingresos a la organización, de forma similar un proyecto sin un plan de despliegue nunca va a ser liberado de forma efectiva a un ambiente productivo.
Principios del Continuous delivery (CD)
8- Mejorar constantemente. No es bueno dejar que un producto software se des actualice ya que este en un futuro no es mantenible, mejora continua significa que un producto software siempre va a estar evolucionando lo que implica que en el tiempo se hace más simple realizar cambio y actualizaciones del mismo.
Continuous delivery (CD), cuenta con 4 buenas practicas
Practicas del Continuous delivery (CD)
1- Construir las versiones solo una vez. En el proceso de despliegue toma demasiado tiempo construir el despliegue entre ambientes, por ende una versión solo se debe de compilar una única vez y esta debe de encontrarse almacenada en donde el mecanismo de despliegue pueda acensar a esta para cada uno de los ambientes.
Practicas del Continuous delivery (CD)
2- Usar el mismo mecanismo de despliegue para los diferentes ambientes. Usar este mecanismo es bueno ya que esta se vuelve una actividad homogénea entre los diferentes ambientes (QA, pruebas, laboratorio, producción).
3- Realizar pruebas de humo sobre el despliegue. No se puede permitir dejar a la suerte el funcionamiento de un despliegue, es necesario realizar pruebas de humo sobre este ya que de esta forma se asegura que este se ejecutó de forma correcta.
Practicas del Continuous delivery (CD)
4- Si uno de los pasos de la puesta a producción de un producto falla se debe de detener todo el proceso. Es una mala práctica realizar parches o ajustes en ambientes productivos cuando un despliegue no satisface las necesidades para su salida, porque si se realiza un despliegue posterior es probable que el error inicial se incluya en la nueva versión, lo que lleva a reprocesos he incrementos en los costos del proyecto.

Daily Meetings...
Cada día de un sprint, se realiza la reunión sobre el estado de un proyecto. Esto se llama daily standup o Stand-up meeting.

Se debe ser puntual para esta reunion, solo hablan los involucrados y debe de tomar 15 a 20 min maximo.
